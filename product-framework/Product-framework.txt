以下是依照 **4 小時版本**重新整理後的完整課程設計。整體原則是：

> **Ground 建立共同視角，Objectives 練習定義成果，Assumptions 練習降低風險，Learn 建立成果回饋閉環。**

課程重點集中在兩個需要實際轉換與判斷的能力：

* **Objectives：從功能需求回推成果目標**
* **Assumptions：從解法中找出關鍵假設**

# 產品工作術：建立數位團隊的產品思維

## GOAL Framework｜4 小時課程規劃

---

# 一、課程整體定位

數位團隊經常接收到大量功能需求，並以時程、範圍與上線作為主要工作目標。然而，功能完成並不必然代表使用者問題獲得改善，也不代表商務成果已經發生。

本課程透過 GOAL Framework，協助學員建立一套從產品方向、成果目標、關鍵假設到成果學習的完整思考方式。

GOAL Framework 包含四個部分：

| GOAL                | 模組          | 核心問題                |
| ------------------- | ----------- | ------------------- |
| **G — Ground**      | 產品工作的全貌     | 我們如何共同創造產品價值？       |
| **O — Objectives**  | 建立成果導向的產品目標 | 我們真正希望改變什麼？         |
| **A — Assumptions** | 驗證關鍵假設      | 我們的解法建立在哪些未被證明的判斷上？ |
| **L — Learn**       | 建立產品學習循環    | 我們如何知道有效，並決定下一步？    |

# 三、課程開場｜從完成工作到創造成果

## 時間

15 分鐘

## 教學目的

讓學員意識到：

* 準時完成，不代表產品成功。
* 功能上線，不代表使用者行為改變。
* 產品團隊需要從交付導向轉向成果導向。
* GOAL Framework 是一套連結方向、決策、執行與學習的工作方式。

## 核心案例

某數位團隊花費六個月完成信用卡優惠推薦功能：

* 如期上線
* 預算符合預期
* 原定功能完成
* 驗收通過

三個月後：

* 使用率偏低
* 客戶仍打電話詢問優惠
* 多數客戶不知道功能存在
* 業務要求增加更多分類與推播

## 開場問題

> 這個產品成功了嗎？

引導學員區分兩種判斷標準：

| 交付標準   | 成果標準     |
| ------ | -------- |
| 是否準時   | 使用者是否採用  |
| 是否符合預算 | 問題是否改善   |
| 功能是否完成 | 行為是否改變   |
| 是否通過驗收 | 商務成果是否出現 |

## 開場結論

> Output 告訴我們完成了什麼；Outcome 告訴我們完成之後，發生了什麼改變。

---

# 四、模組一｜Ground：產品工作的全貌

## 時間

35 分鐘

## 模組定位

快速建立共同產品視角，讓學員理解產品工作不是單一部門或單一流程，而是方向、協作、執行與成果之間的連結。

本模組的目的不是深入教授產品生命週期，而是建立後續課程所需的共同地圖。

## 學習目標

完成本模組後，學員能夠：

1. 區分產品交付與產品成果。
2. 說明產品工作的四個面向。
3. 看見不同職能在產品決策中的價值。
4. 辨識產品工作中常見的斷點。
5. 理解產品工作是一個持續學習的循環。

## 核心架構

| 面向                | 核心問題       | 常見斷點            |
| ----------------- | ---------- | --------------- |
| **Direction**     | 我們為什麼做？    | 只收到功能，不知道目標     |
| **Collaboration** | 誰要一起做判斷？   | 角色依序交接，重要判斷太晚發生 |
| **Execution**     | 如何把方向變成行動？ | 需求提出後直接進入開發     |
| **Outcome**       | 如何知道有效？    | 上線後只看系統是否正常     |

## 產品工作閉環

> Direction → Collaboration → Execution → Outcome → 回到新的 Direction

本模組需要強調：

* 團隊通常不是完全缺少其中一個面向。
* 真正的問題常發生在面向之間的連結。
* 有策略，不代表策略有影響需求排序。
* 有數據，不代表數據有影響下一步決策。
* 有法遵與技術參與，不代表判斷發生得夠早。

## 簡化產品生命週期

> 理解問題 → 設定目標 → 驗證解法 → 建置交付 → 觀察成果

教學重點：

> 每一個階段都在降低不同的不確定性。

## 跨職能角色的判斷價值

| 角色    | 提供的主要判斷     |
| ----- | ----------- |
| 商務    | 為何值得投入？     |
| 產品    | 現在最重要的是什麼？  |
| UX／研究 | 使用者真正遇到什麼？  |
| 技術    | 是否可行且可持續？   |
| 數據    | 如何知道有效？     |
| 營運／客服 | 真實使用情境是什麼？  |
| 法遵／風控 | 哪些限制需要提早納入？ |

本段不要教成角色分工，而要強調：

> 跨職能協作的價值，是讓重要判斷提早發生。

## 活動：快速斷點診斷

提供信用卡優惠推薦案例，請小組選出最重要的斷點：

* Direction
* Collaboration
* Execution
* Outcome
* 面向之間的連結

完成句型：

> 我們認為最重要的斷點是＿＿＿＿，因為＿＿＿＿。

## 模組產出

一個清楚的產品工作問題診斷：

* 問題發生在哪一個面向
* 或發生在哪兩個面向的連結
* 對產品造成什麼影響

## 模組收束

四個檢查問題：

1. 我們是否知道為什麼做？
2. 重要角色是否共同參與判斷？
3. 工作方式是否支持快速學習？
4. 上線後是否能看見成果？

## 銜接模組二

> 如果團隊不知道希望創造什麼成果，就無法判斷哪些工作值得做。

# 五、模組二｜Objectives：建立成果導向的產品目標

## 時間

65 分鐘

## 模組定位

這是第一個主要練習模組。

核心能力不是撰寫漂亮的目標句子，而是能夠把「功能需求」往回拆解，找出：

* 商務為什麼需要這件事
* 使用者需要發生什麼改變
* 團隊真正要追求的產品成果
* 功能只是眾多可能解法之一

## 學習目標

完成本模組後，學員能夠：

1. 區分 Business Objective、Product Objective、Outcome 與 Output。
2. 從功能需求回推商務目的。
3. 找出需要改變的使用者行為。
4. 把功能句子改寫成成果導向的產品目標。
5. 使用產品目標支持需求分析與功能取捨。

## 核心轉換路徑

> 功能需求
> ↓
> 使用者正在遇到什麼問題？
> ↓
> 希望使用者產生什麼行為改變？
> ↓
> 這個改變支持什麼商務成果？
> ↓
> 形成產品目標

## 四層目標關係

| 層次                 | 核心問題        | 範例              |
| ------------------ | ----------- | --------------- |
| Business Objective | 組織為何要投入？    | 提高信用卡活躍度        |
| User Outcome       | 使用者要發生什麼改變？ | 更容易找到適合自己的優惠    |
| Product Objective  | 產品希望促成什麼行為？ | 提高消費前查詢與使用優惠的比例 |
| Product Output     | 團隊可能交付什麼？   | 推薦、搜尋、提醒、比較功能   |

## 七大 Business Objectives

| 類型          | 目的        |
| ----------- | --------- |
| Acquisition | 取得新客戶     |
| Activation  | 促成首次成功使用  |
| Engagement  | 增加使用與互動   |
| Retention   | 提高持續使用與留存 |
| Revenue     | 增加收入或交易價值 |
| Efficiency  | 降低成本與作業負擔 |
| Risk／Trust  | 降低風險並提高信任 |

七大目標的功能是幫助學員判斷：

> 這項需求主要想改善哪一種商務成果？

一個需求可能影響多個目標，但課堂中要求學員選擇一個主要目標，避免所有目標都重要。

## 產品目標句型

> 我們希望協助【目標使用者】，
> 在【關鍵情境】中，
> 從【目前行為或狀態】，
> 改變為【期望行為或狀態】，
> 進而支持【商務目標】。

## 練習一：辨識 Output 與 Outcome

讓學員快速判斷：

* 完成線上調額功能
* 降低客戶致電客服申請調額的比例
* 上線優惠推薦專區
* 提高客戶使用指定優惠的比例
* 完成新版登入流程
* 降低登入失敗率

目的不是考試，而是讓學員看見團隊習慣使用 Output 描述目標。

## 練習二：功能需求逆向推導

每組取得一個功能需求，例如：

* 建立 AI 客服
* 新增優惠推薦
* 增加即時通知
* 建立會員等級
* 優化線上申請流程
* 新增交易儀表板

依序回答：

1. 這個功能主要支持哪一類 Business Objective？
2. 目標使用者是誰？
3. 使用者目前在什麼情境中遇到問題？
4. 希望改變什麼使用者行為？
5. 什麼 Outcome 可以代表改善？
6. 除了原本功能，還有哪些可能解法？

## 練習三：改寫產品目標

將原始功能需求改寫為成果導向目標。

例如，原始需求：

> 建立信用卡優惠推薦功能。

改寫後：

> 協助持有多張信用卡的客戶，在準備消費時快速判斷適用優惠，提高客戶在消費前查詢與使用優惠的比例，進而提升信用卡活躍度。

## 功能取捨原則

當不同功能競爭資源時，使用四個問題：

1. 與產品目標的連結有多強？
2. 能影響多少目標使用者？
3. 預期能帶來多大的行為改變？
4. 目前有多少證據支持這個判斷？

本課程不需要導入複雜評分模型，重點是建立共同取捨依據。

## 模組產出

每組完成一張產品目標卡：

* 原始功能需求
* Business Objective
* 目標使用者
* 關鍵情境
* 目前行為
* 期望行為
* Product Objective
* Outcome
* 可能解法

## 模組收束

> 功能是解法，產品目標描述的是我們希望促成的改變。

## 銜接模組三

> 當我們提出一個解法時，也同時做出了許多尚未被證明的判斷。

# 六、模組三｜Assumptions：建立實驗思維，驗證關鍵假設

## 時間

65 分鐘

## 模組定位

這是第二個主要練習模組。

學員需要從「討論解法好不好」轉向「辨識我們正在假設什麼」。

本模組的核心不是介紹大量實驗工具，而是建立三個能力：

1. 找出解法背後的假設
2. 判斷哪一個假設應優先驗證
3. 選擇低成本的證據取得方式

## 學習目標

完成本模組後，學員能夠：

1. 看見需求與解法背後的假設。
2. 區分不同類型的產品風險。
3. 找出高影響、高不確定性的假設。
4. 選擇適合的驗證方式。
5. 設定可支持決策的判斷標準。

## 四類產品假設

| 假設類型        | 核心問題              |
| ----------- | ----------------- |
| Value       | 使用者需要或願意使用嗎？      |
| Usability   | 使用者能理解並順利操作嗎？     |
| Feasibility | 技術與資料能支援嗎？        |
| Viability   | 商務、營運、法遵與風險上能成立嗎？ |

## 假設拆解示例

解法：

> 系統依據客戶消費紀錄，自動推薦最適合的信用卡優惠。

可能假設：

* 客戶需要個人化推薦。
* 客戶願意讓系統使用交易資料。
* 推薦比自行搜尋更有效率。
* 系統能正確理解優惠條件。
* 客戶看得懂推薦理由。
* 推薦內容不會造成誤導。
* 法遵允許這種推薦方式。
* 推薦會提高優惠使用率或刷卡金額。

核心觀念：

> 寫進需求文件的內容，仍然可能只是尚未被證明的假設。

## 假設優先排序

使用兩個主要維度：

* 影響程度
* 不確定程度

|     | 低不確定性 | 高不確定性 |
| --- | ----- | ----- |
| 高影響 | 持續確認  | 優先驗證  |
| 低影響 | 延後處理  | 暫不投入  |

必要時再加入第三個判斷：

* 驗證成本

課堂中不需要計算分數，只要能解釋為什麼某一個假設值得先驗證。

## 驗證方式

| 方法        | 適合回答的問題        |
| --------- | -------------- |
| Prototype | 使用者是否理解、願意使用？  |
| MVP       | 真實情境中是否產生預期行為？ |
| Spike     | 技術或資料是否可行？     |
| 人工試行      | 營運模式是否成立？      |
| 訪談／測試     | 使用者需求與理解是否正確？  |
| 數據分析      | 問題是否存在、行為是否相關？ |

## MVP 的教學定位

避免把 MVP 教成「功能較少的正式產品」。

本課程採用的定義：

> MVP 是能以最低合理成本取得關鍵證據的產品形式。

## 實驗設計五要素

| 元素   | 核心問題          |
| ---- | ------------- |
| 假設   | 我們相信什麼？       |
| 對象   | 要向誰驗證？        |
| 方法   | 如何取得證據？       |
| 判斷標準 | 看到什麼結果代表獲得支持？ |
| 決策   | 不同結果分別採取什麼行動？ |

## 實驗句型

> 我們相信【目標使用者】在【情境】中會【期望行為】。
> 我們將透過【驗證方式】取得證據。
> 當【觀察指標】達到【判斷標準】時，代表假設獲得初步支持。
> 根據結果，我們將決定【擴大、調整、再次驗證或停止】。

## 主要活動：從解法找出關鍵假設

延續模組二的產品目標，完成以下步驟：

### 步驟一：提出一個可能解法

例如：

* 個人化推薦
* 情境搜尋
* 優惠比較
* 到期提醒
* 消費前提示
* 客服協助

### 步驟二：列出假設

至少列出：

* 兩個 Value Assumptions
* 一個 Usability Assumption
* 一個 Feasibility Assumption
* 一個 Viability Assumption

### 步驟三：選出關鍵假設

判斷：

* 如果這個假設錯了，影響有多大？
* 團隊目前有多少證據？
* 是否能以低成本取得更多證據？

### 步驟四：選擇驗證方式

從 Prototype、MVP、Spike、訪談、人工試行或數據分析中選擇。

### 步驟五：設定判斷標準

例如：

* 10 位受測者中至少 8 位能自行完成任務。
* 推薦點擊率高於既有入口。
* 技術測試能在兩秒內完成推薦。
* 法遵確認推薦說明符合揭露要求。

## 模組產出

每組完成一張假設與實驗卡：

* Product Objective
* 解法構想
* 四類假設
* 關鍵假設
* 驗證方式
* 判斷標準
* 可能決策

## 模組收束

> 團隊不需要一開始知道所有答案，但需要知道哪一個問題最值得先回答。

## 銜接模組四

> 驗證完成後，真正重要的是如何把證據轉化為產品決策。

# 七、模組四｜Learn：建立產品學習循環

## 時間

35 分鐘

## 模組定位

建立產品工作閉環，讓學員理解上線或實驗完成後，需要透過數據、行為與回饋決定下一步。

本模組不深入教授分析方法或指標系統，而是建立成果衡量與決策的基本結構。

## 學習目標

完成本模組後，學員能夠：

1. 區分 Activity、Output 與 Outcome。
2. 理解行為成果與商務成果的關係。
3. 同時使用量化與質化證據。
4. 根據證據做出產品決策。
5. 建立簡單的產品學習循環。

## 四層成果衡量

| 層次               | 核心問題       | 範例           |
| ---------------- | ---------- | ------------ |
| Activity         | 團隊做了多少工作？  | 訪談 20 位客戶    |
| Output           | 團隊交付了什麼？   | 上線優惠推薦功能     |
| Behavior Outcome | 使用者行為是否改變？ | 更多客戶在消費前查看優惠 |
| Business Outcome | 商務結果是否改善？  | 優惠使用率與刷卡金額提高 |

## 領先與落後指標

| 類型   | 說明              | 範例          |
| ---- | --------------- | ----------- |
| 領先指標 | 較早反映行為是否朝正確方向改變 | 查看率、點擊率、完成率 |
| 落後指標 | 最終商務成果          | 營收、交易額、留存率  |

教學重點：

* 只看落後指標，可能太晚才發現方向錯誤。
* 只看領先指標，也無法證明最終商務價值。
* 團隊需要理解兩者之間的關係。

## 量化與質化證據

| 量化資料      | 質化資料       |
| --------- | ---------- |
| 發生了什麼？    | 為什麼發生？     |
| 哪裡出現流失？   | 使用者為什麼放棄？  |
| 哪一群人反應較好？ | 這一群人有什麼不同？ |
| 行為是否改變？   | 使用者如何理解產品？ |

## 產品決策四種結果

| 決策         | 適用情況           |
| ---------- | -------------- |
| Maintain   | 結果符合預期，持續維持    |
| Improve    | 方向正確，但效果需要提升   |
| Scale      | 證據足夠，擴大客群或投入   |
| Stop／Pivot | 假設未獲支持，停止或改變方向 |

## 快速活動：讀取產品結果

提供一組模擬資料：

* 優惠推薦曝光：50,000
* 點擊推薦：8,000
* 查看詳情：4,500
* 登錄優惠：1,200
* 完成指定消費：350
* 客服詢問增加 15%
* 客戶反映看不懂推薦原因

請學員回答：

1. 哪些是 Activity、Output 或 Outcome？
2. 使用者在哪個階段大量流失？
3. 目前資料能支持什麼結論？
4. 還需要哪些質化資料？
5. 下一步應 Maintain、Improve、Scale 或 Stop？

## 產品學習循環

> Observe → Interpret → Question → Experiment → Decide → Adapt

| 步驟         | 團隊工作         |
| ---------- | ------------ |
| Observe    | 觀察數據與回饋      |
| Interpret  | 理解可能原因       |
| Question   | 提出下一個問題      |
| Experiment | 設計驗證方式       |
| Decide     | 做出產品決策       |
| Adapt      | 調整目標、解法或優先順序 |

# 八、整合與行動｜串聯 GOAL Framework

## 時間

15 分鐘

## 整合目的

讓學員重新看見四個模組之間的邏輯：

| GOAL        | 最終問題             |
| ----------- | ---------------- |
| Ground      | 我們是否擁有共同的產品工作視角？ |
| Objectives  | 我們希望創造什麼成果？      |
| Assumptions | 我們必須先證明什麼？       |
| Learn       | 我們如何根據結果決定下一步？   |

## GOAL 快速檢核

請學員選擇目前工作中的一項需求，回答：

### Ground

* 這項工作目前最明顯的斷點是什麼？

### Objectives

* 它真正希望改變什麼使用者行為？
* 支持什麼商務成果？

### Assumptions

* 目前最關鍵且最不確定的假設是什麼？
* 可以如何取得證據？

### Learn

* 上線或驗證後需要觀察什麼？
* 哪些結果會讓團隊改變方向？

## 個人行動卡

請每位學員完成：

> 未來兩週內，我會選擇一項正在進行的產品需求，使用 GOAL Framework 重新檢視產品目標、關鍵假設與成果指標，並提出一項可以立即執行的驗證行動。

# 十、4 小時版本的核心學習成果

課程結束後，學員應能完成以下四個動作：

1. **看全貌**
   使用 Direction、Collaboration、Execution、Outcome 看見產品工作斷點。

2. **找成果**
   將功能需求回推為使用者行為與商務成果。

3. **拆假設**
   從解法中找出 Value、Usability、Feasibility 與 Viability Assumptions。

4. **做決策**
   根據數據與回饋決定維持、改善、擴大或停止。

整門課最重要的轉換是：

> 從「這個功能要怎麼做」，轉向「我們希望改變什麼、正在假設什麼，以及需要取得什麼證據」。

這個版本已經可以作為後續設計四個模組投影片、講師手冊與學員畫布的共同母版。
